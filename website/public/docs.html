Installation & Upgrades

Run one command in your terminal. The installer downloads the server, installs dependencies, and registers with Cursor automatically.

curl https://vortexado.vercel.app/install -fsS | bash

After install, restart Cursor IDE. You'll see ado-testforge in your MCP server list.

Same command for upgrades. Running the install command on an existing installation automatically detects the previous version, backs up your credentials, downloads the latest release, restores credentials, and re-installs dependencies. Your setup is preserved — just restart Cursor after the upgrade completes.

Uninstall

To remove VortexADO completely (MCP config entry + installed files):

curl https://vortexado.vercel.app/uninstall -fsS | bash

Credentials Setup

After install, configure your credentials. You have two options:

Option 1: Configuration UI (recommended)

Type /ado-connect in Cursor's chat. This opens a web interface where you can enter credentials and test connections in real time.

Creating Your ADO Personal Access Token

The MCP server connects to Azure DevOps on your behalf using a Personal Access Token (PAT). You can create one with Full access or with Custom defined scopes for tighter security.

Steps

  1. Open your browser and go to: https://dev.azure.com/{your-org}/_usersSettings/tokens
  2. Click + New Token
  3. Set a name (e.g. "VortexADO MCP") and expiration (90 days recommended)
  4. Choose Full access for simplicity, or Custom defined for minimum-privilege access (see table below)
  5. Click Create and copy the token immediately — you won't see it again

Custom defined scopes (minimum required)

If you prefer least-privilege access, select Custom defined and enable these scopes:

Scope (UI Label)Access LevelWhy it's needed
Work ItemsRead & WriteRead user stories, create/update/delete test cases
Test ManagementRead & WriteManage test plans, suites, and test case associations
Project and TeamRead & WriteResolve project context, area paths, and iteration paths
NotificationsReadAccess user notification preferences for work item context
Service ConnectionsReadResolve service endpoints referenced in work items
SymbolsRead & WriteAccess symbol information for attachment handling
User ProfileRead & WriteResolve user identities for assigned-to fields
Secure FilesReadAccess secure file references in test configurations
Tip: If you only need basic test case drafting and publishing, the two essential scopes are Work Items (Read & Write) and Test Management (Read & Write). The additional scopes enable full feature coverage including suite management, user resolution, and attachment handling.
Security note: Never paste your PAT in Cursor's chat, email, or shared documents. Store it in a password manager. If compromised, revoke it immediately in ADO and create a new one.

Creating Your Confluence API Token

If your User Stories link to Solution Design documents in Confluence, VortexADO can automatically fetch that content to enrich test case generation. This requires an Atlassian API token.

Steps

  1. Go to id.atlassian.com/manage-profile/security/api-tokens
  2. Click Create API token
  3. Set a label (e.g. "VortexADO MCP") and click Create
  4. Copy the token immediately

For Confluence Cloud with OAuth 2.0 apps or fine-grained tokens, these are the minimum scopes required:

Required Confluence scopes

ScopePurpose
read:confluence-content.allRead full page content (storage format HTML)
read:confluence-content.summaryRead page summaries and metadata
read:confluence-space.summaryList and resolve spaces
read:page:confluenceRead individual pages by ID
read:content:confluenceRead content objects (pages, blogs, comments)
read:content-details:confluenceRead expanded content details (body, version, ancestors)
read:space:confluenceRead space details and permissions context
read:attachment.download:confluenceDownload page attachments (images, diagrams)
Note: Standard Atlassian API tokens (email + token as basic auth) work out of the box for most Confluence Cloud instances without needing to configure individual scopes. The scopes above apply only if your organization uses OAuth 2.0 apps or Forge apps with fine-grained permission models.

If Confluence is not configured, VortexADO works normally — Solution Design content will simply not be available during test case drafting, and you can paste relevant content manually when the agent asks.

Team Configuration

VortexADO adapts to your team's conventions via a single configuration file placed alongside the installation:

~/.ado-testforge-mcp/conventions.config.json

This file controls everything from test case title format to suite hierarchy rules. Here's what you can configure:

Test case titles

Define how test case titles are structured. Set a prefix (e.g. "TC"), separator, number padding, and maximum length. The template uses placeholders like user story ID, test case number, feature tags, and summary.

Prerequisites format

Configure the sections that appear in test case prerequisites: Persona, Pre-requisite conditions, and Test Data. You can set which are required and control the formatting style (technical condition-based format like Object.Field = Value or narrative style).

Default personas

Define your team's standard test personas — their labels, profiles, roles, and permission groups. These are automatically included in generated test cases. The column headings for roles and permission groups are customizable to match your organization's terminology.

Suite hierarchy

Control how the MCP organizes test suites in your test plan:

  • Sprint prefix — how sprint folders are named (e.g. "Sprint_", "Iteration_")
  • Separator — between IDs and titles in folder names
  • Non-epic folder — name for the catch-all folder for user stories without a parent epic
  • Test plan mapping — which area paths route to which test plans (supports multiple plans)
  • Title prefix — the prefix used in query-based suite WIQL queries (default "TC")

Solution Design integration

Configure which ADO field contains Confluence links and define usage rules for how Solution Design content informs test case generation (what to use for business context, what to ignore like code snippets).

Additional context fields

Register custom ADO fields (like Impact Assessment or Reference Documentation) that should be fetched for rich content and links. Each field can independently enable or disable link/image extraction.

Image handling

VortexADO can extract and process images embedded in ADO rich-text fields and Confluence page attachments — wireframes, flow diagrams, UI mockups. This is controlled by the images section in your configuration.

ON/OFF toggles

SettingDefaultWhat it does
enabledONMaster switch. When OFF, no images are fetched or processed at all.
returnMcpImagePartsOFFWhen ON, sends actual image pixel data to the AI agent (requires a vision-capable model). When OFF, only image metadata (URL, filename, alt text) is included — the agent can still reference images by link.
saveLocallyOFFWhen ON, saves downloaded images to your workspace alongside the draft files. Useful for offline review or attaching images to documentation.
inlineSvgAsTextONWhen ON, SVG diagrams are delivered as both an image part and inline XML text, allowing agents to read element IDs, colors, and labels directly.

Size and quality guardrails

SettingDefaultWhat it does
maxPerUserStory20Maximum number of images processed per user story fetch
maxBytesPerImage2 MBImages larger than this are downscaled before processing
maxTotalBytesPerResponse4 MBAggregate budget — once reached, remaining images are listed as links only (prevents context overflow)
minBytesToKeep4 KBImages smaller than this are skipped (likely icons or tracking pixels)
downscaleLongSidePx1600 pxOversized images are downscaled so the longest side is this many pixels
downscaleQuality85JPEG quality after downscaling (1–100)
mimeAllowlistPNG, JPEG, GIF, SVGOnly these image types are processed; others are skipped

Save path

When saveLocally is ON, images are saved to a folder alongside the test case draft. The default path template is tc-drafts/US_{usId}/attachments. If the workspace is read-only, the tool falls back to MCP image parts without failing.

Recommended for most teams: Leave enabled: true and returnMcpImageParts: false (the defaults). This way the agent receives image metadata and links without consuming vision-model tokens. Turn returnMcpImageParts ON only when you need the AI to interpret diagram content or screenshot details directly.

Context budgets

Control how much external content is fetched per user story:

SettingDefaultWhat it does
maxConfluencePagesPerUserStory10Maximum number of linked Confluence pages to fetch. Excess links are surfaced as "unfetched" with clickable URLs.
maxTotalFetchSeconds45sTotal time budget for all external fetches (Confluence pages + images). Prevents slow responses on user stories with many links.

All-fields pass-through

By default, VortexADO returns every populated field on a user story (with system noise filtered out). You can control this behavior:

  • passThrough (ON by default) — include all populated ADO fields beyond the primary ones
  • omitSystemNoise (ON by default) — filter out revision metadata, watermarks, and internal tracking fields
  • omitExtraRefs — list specific field reference names to exclude from the pass-through
The configuration file is optional. Defaults work for most teams. Start without it and add customization as needed. When upgrading, note that the installer overwrites this file — keep a backup of your customizations.

Verify Your Setup

After configuring credentials, type /ado-check in Cursor's chat. This runs a diagnostic that tests:

  • ADO connection and PAT validity
  • Project access and permissions
  • Confluence connectivity (if configured)
  • Configuration file presence and validity

The result shows a clear verdict (ready / degraded / broken) with exact next steps for any issues.

Your First Test Case

Once /ado-check shows "ready", here's the workflow:

  1. Draft — Type /qa-draft and provide a User Story ID. The MCP reads the full story context (fields, linked pages, images) and generates a test case draft with solution summary and QA cheat sheet in your workspace.
  2. Review — Open the generated files in your editor. Edit test cases, adjust coverage, fix any assumptions. The draft is yours to shape.
  3. Publish — Type /qa-publish with the same User Story ID. The MCP checks for duplicates, asks for confirmation, then creates the test cases in ADO and auto-organizes them into the correct suite hierarchy.

Troubleshooting

MCP server not showing in Cursor

Restart Cursor after install. Check that ~/.cursor/mcp.json contains an ado-testforge entry.

ADO connection fails

Verify your PAT hasn't expired. Check that you have the correct org and project names. Run /ado-check for a detailed diagnostic.

Confluence pages not loading

Ensure your Confluence API token is valid and the email matches. The system tries the direct URL first, then falls back to the Atlassian Cloud API automatically.
