Documentation & Setup Guide

Everything you need to install, configure, and start using VortexADO.

Prerequisites

RequirementDetails
Node.jsVersion 18 or higher
Operating SystemmacOS, Linux, or Windows (WSL)
Cursor IDEAny recent version with MCP support
Azure DevOpsPersonal Access Token with Work Items + Test Management scopes
Confluence (optional)API token for Solution Design page fetching

Installation

Run one command in your terminal. The installer downloads the server, installs dependencies, and registers with Cursor automatically.

curl https://vortexado.vercel.app/install -fsS | bash

After install, restart Cursor IDE. You'll see ado-testforge in your MCP server list.

Uninstall

To remove VortexADO completely (MCP config entry + installed files):

curl https://vortexado.vercel.app/uninstall -fsS | bash

Credentials Setup

After install, configure your credentials. You have two options:

Option 1: Configuration UI (recommended)

Type /ado-connect in Cursor's chat. This opens a web interface where you can enter credentials and test connections in real time.

Option 2: Manual edit

Edit the credentials file directly:

~/.ado-testforge-mcp/credentials.json

Required fields

FieldDescriptionWhere to get it
ado_patAzure DevOps Personal Access TokenADO → User Settings → Personal Access Tokens
ado_orgOrganization nameThe part after dev.azure.com/
ado_projectProject nameYour ADO project

Optional fields (Confluence)

FieldDescription
confluence_base_urlYour Confluence instance URL (e.g. https://yourteam.atlassian.net/wiki)
confluence_emailAtlassian account email
confluence_api_tokenAPI token from id.atlassian.com/manage-profile/security/api-tokens
Your PAT needs these scopes: Work Items (Read & Write) and Test Management (Read & Write). For permanent deletion, you also need Project Administrator permissions.

Team Configuration

VortexADO adapts to your team's conventions via a single configuration file placed alongside the installation:

~/.ado-testforge-mcp/conventions.config.json

This file controls everything from test case title format to suite hierarchy rules. Here's what you can configure:

Test case titles

Define how test case titles are structured. Set a prefix (e.g. "TC"), separator, number padding, and maximum length. The template uses placeholders like user story ID, test case number, feature tags, and summary.

Prerequisites format

Configure the sections that appear in test case prerequisites: Persona, Pre-requisite conditions, and Test Data. You can set which are required and control the formatting style (technical condition-based format like Object.Field = Value or narrative style).

Default personas

Define your team's standard test personas — their labels, profiles, roles, and permission groups. These are automatically included in generated test cases. The column headings for roles and permission groups are customizable to match your organization's terminology.

Suite hierarchy

Control how the MCP organizes test suites in your test plan:

  • Sprint prefix — how sprint folders are named (e.g. "Sprint_", "Iteration_")
  • Separator — between IDs and titles in folder names
  • Non-epic folder — name for the catch-all folder for user stories without a parent epic
  • Test plan mapping — which area paths route to which test plans (supports multiple plans)
  • Title prefix — the prefix used in query-based suite WIQL queries (default "TC")

Solution Design integration

Configure which ADO field contains Confluence links and define usage rules for how Solution Design content informs test case generation (what to use for business context, what to ignore like code snippets).

Additional context fields

Register custom ADO fields (like Impact Assessment or Reference Documentation) that should be fetched for rich content and links. Each field can independently enable or disable link/image extraction.

Image handling

Control embedded image behavior: enable/disable, size limits, downscale resolution, MIME type allowlist, whether to save locally or return as MCP content parts, and a total response budget to prevent context overflow.

The configuration file is optional. Defaults work for most teams. Start without it and add customization as needed.

Verify Your Setup

After configuring credentials, type /ado-check in Cursor's chat. This runs a diagnostic that tests:

  • ADO connection and PAT validity
  • Project access and permissions
  • Confluence connectivity (if configured)
  • Configuration file presence and validity

The result shows a clear verdict (ready / degraded / broken) with exact next steps for any issues.

Your First Test Case

Once /ado-check shows "ready", here's the workflow:

  1. Draft — Type /qa-draft and provide a User Story ID. The MCP reads the full story context (fields, linked pages, images) and generates a test case draft with solution summary and QA cheat sheet in your workspace.
  2. Review — Open the generated files in your editor. Edit test cases, adjust coverage, fix any assumptions. The draft is yours to shape.
  3. Publish — Type /qa-publish with the same User Story ID. The MCP checks for duplicates, asks for confirmation, then creates the test cases in ADO and auto-organizes them into the correct suite hierarchy.

Updating

Run the install command again — it detects the existing installation, backs up your credentials, downloads the latest version, and restores credentials automatically.

curl https://vortexado.vercel.app/install -fsS | bash

Troubleshooting

MCP server not showing in Cursor

Restart Cursor after install. Check that ~/.cursor/mcp.json contains an ado-testforge entry.

ADO connection fails

Verify your PAT hasn't expired. Check that you have the correct org and project names. Run /ado-check for a detailed diagnostic.

Confluence pages not loading

Ensure your Confluence API token is valid and the email matches. The system tries the direct URL first, then falls back to the Atlassian Cloud API automatically.
