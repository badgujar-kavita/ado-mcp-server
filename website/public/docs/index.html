Installation & Upgrades

Run one command in your terminal. The installer downloads the server, installs dependencies, and registers with Cursor automatically.

curl https://vortexado.vercel.app/install -fsS | bash

After install, restart Cursor IDE. You'll see vortex-ado in your MCP server list.

Same command for upgrades. Running the install command on an existing installation automatically detects the previous version, backs up your credentials, downloads the latest release, restores credentials, and re-installs dependencies. Your setup is preserved — just restart the MCP server (Settings → MCP → refresh vortex-ado) after the upgrade completes.

Uninstall

To remove Vortex ADO completely (MCP config entry + installed files):

curl https://vortexado.vercel.app/uninstall -fsS | bash

Credentials Setup

After install, open your project folder in Cursor and configure credentials per-workspace. The wizard writes connection details to <workspace>/.vortex-ado/config.json and stores your PAT in the operating system's secure credential store — never on disk.

Multi-project parallel work. Two Cursor windows on two different ADO projects each get their own per-workspace config and their own keychain entry. They never share state. See Per-Workspace Conventions below.

Option 1: Configuration UI (recommended)

Open your project folder in Cursor, then type /ado-connect in the agent chat. The wizard:

  • Lets you enter credentials and test connections in real time
  • Writes connection details (org, project, URL, optional Confluence) to <workspace>/.vortex-ado/config.json
  • Stores the PAT in the OS keychain under service vortex-ado, account ado::{org}::{project}
  • Refuses to write into your home directory or a non-writable cwd (safety guard)
  • On re-run, preserves your manual edits to non-credential blocks (testCaseTitle, personas, testPlanMapping)

Option 2: Manual edit

Create <workspace>/.vortex-ado/config.json by hand:

{
  "version": 1,
  "ado": {
    "url": "https://dev.azure.com/YourOrg",
    "org": "YourOrg",
    "project": "Your Project"
  }
}

Then store your PAT in the OS keychain under service vortex-ado, account ado::YourOrg::Your Project. See the full reference for per-platform commands.

Creating Your ADO Personal Access Token

The MCP server connects to Azure DevOps on your behalf using a Personal Access Token (PAT). You can create one with Full access or with Custom defined scopes for tighter security.

Steps

  1. Open your browser and go to: https://dev.azure.com/{your-org}/_usersSettings/tokens
  2. Click + New Token
  3. Set a name (e.g. "Vortex ADO MCP") and expiration (90 days recommended)
  4. Choose Full access for simplicity, or Custom defined for minimum-privilege access (see table below)
  5. Click Create and copy the token immediately — you won't see it again

Custom defined scopes (minimum required)

If you prefer least-privilege access, select Custom defined and enable these scopes:

Scope (UI Label)Access LevelWhy it's needed
Work ItemsRead & WriteRead user stories, create/update/delete test cases
Test ManagementRead & WriteManage test plans, suites, and test case associations
Project and TeamRead & WriteResolve project context, area paths, and iteration paths
NotificationsReadAccess user notification preferences for work item context
Service ConnectionsReadResolve service endpoints referenced in work items
SymbolsRead & WriteAccess symbol information for attachment handling
User ProfileRead & WriteResolve user identities for assigned-to fields
Secure FilesReadAccess secure file references in test configurations
Tip: If you only need basic test case drafting and publishing, the two essential scopes are Work Items (Read & Write) and Test Management (Read & Write). The additional scopes enable full feature coverage including suite management, user resolution, and attachment handling.
Security note: Never paste your PAT in Cursor's chat, email, or shared documents. Store it in a password manager. If compromised, revoke it immediately in ADO and create a new one.

Creating Your Confluence API Token

If your User Stories link to Solution Design documents in Confluence, Vortex ADO can automatically fetch that content to enrich test case generation. This requires an Atlassian API token.

Steps

  1. Go to id.atlassian.com/manage-profile/security/api-tokens
  2. Click Create API token
  3. Set a label (e.g. "Vortex ADO MCP") and click Create
  4. Copy the token immediately

Minimum scopes (if creating a token with custom scopes)

If your organization uses fine-grained access tokens, ensure at least these scopes are enabled:

ScopePurpose
read:confluence-content.allRead full page content (storage format HTML)
read:confluence-content.summaryRead page summaries and metadata
read:confluence-space.summaryList and resolve spaces
read:page:confluenceRead individual pages by ID
read:content:confluenceRead content objects (pages, blogs, comments)
read:content-details:confluenceRead expanded content details (body, version, ancestors)
read:space:confluenceRead space details and permissions context
read:attachment.download:confluenceDownload page attachments (images, diagrams)

If Confluence is not configured, Vortex ADO works normally — Solution Design content will simply not be available during test case drafting, and you can paste relevant content manually when the agent asks.

Team Configuration

Vortex ADO adapts to your team's conventions via a per-workspace configuration file:

<workspace>/.vortex-ado/config.json

This file controls everything from test case title format to suite hierarchy rules. The MCP merges your workspace overlay on top of framework defaults — anything you don't specify uses the default. Settings are grouped into logical sections:

See Per-Workspace Conventions below for the edit-priority guide and multi-project scenario, or open docs/conventions.md for the full annotated reference.

Recommended for most teams: Tick "Enable image fetching" in the /ado-connect wizard if your User Stories carry meaningful screenshots or diagrams the agent should see while drafting test cases. Otherwise leave it off — image processing adds time and bytes to every /qa-draft run.

Test Case Format

Controls how generated test cases are structured and named.

  • Title template — define the prefix (e.g. "TC"), separator, number padding, and maximum length. The template uses placeholders like user story ID, test case number, feature tags, and summary.
  • Prerequisites sections — configure which sections appear (Persona, Pre-requisite conditions, Test Data), which are required, and the formatting style (technical condition-based like Object.Field = Value, or narrative).
  • Default personas — define your team's standard test personas with their labels, profiles, roles, and permission groups. These are automatically included in generated test cases. Column headings for roles and groups are customizable to match your organization's terminology.

Suite Hierarchy

Controls how the MCP organizes test suites within your test plans.

  • Sprint prefix — how sprint folders are named (e.g. "Sprint_", "Iteration_")
  • Separator — character(s) between IDs and titles in folder names
  • Non-epic folder — name for the catch-all folder for user stories without a parent epic
  • Test plan mapping — which area paths route to which test plans (supports multiple plans per project)
  • Title prefix — the prefix used in query-based suite queries (default "TC")

Context & Enrichment

Controls what information the MCP fetches when reading a User Story.

  • Solution Design integration — configure which ADO field contains Confluence links, and define usage rules for how Solution Design content informs test case generation (what to use for business context, what to ignore like code snippets).
  • Additional context fields — register custom ADO fields (like Impact Assessment or Reference Documentation) that should be fetched for rich content and links. Each field can independently enable or disable link and image extraction.

Fetching & Image Processing

Vortex ADO can extract and process images embedded in ADO rich-text fields and Confluence page attachments — wireframes, flow diagrams, UI mockups. The AI agent uses these images as visual reference when designing test cases.

Image fetching is OFF by default. Turn it on with the "Enable image fetching" checkbox in Tab 2 of the /ado-connect wizard. That single toggle is the only image setting tenants need to think about — every other image budget (size caps, downscale quality, MIME allowlist) stays at framework defaults that work for most teams.

Per-Workspace Conventions

As of Phase 1, all per-tenant configuration lives per-workspace at:

<workspace>/.vortex-ado/config.json

This unblocks the multi-project use case: a QA engineer working on Project_ABC in one Cursor window and Project_XYZ in another now gets two independent configs and two independent keychain entries — no more shared global state. Each Cursor window's MCP process reads the config from its own working directory.

Connection details (org, project, URL, Confluence) and team conventions (test case naming, personas, plan mappings) all live in this one file. The MCP merges your overlay on top of framework defaults, so you only specify what's project-specific.

For the canonical reference — full annotated schema, copy-pasteable starter template, and per-platform keychain commands — open docs/conventions.md in the repo.

Verify Your Setup

After configuring credentials, type /ado-check in Cursor's chat. This runs a diagnostic that tests:

  • ADO connection and PAT validity
  • Project access and permissions
  • Confluence connectivity (if configured)
  • Configuration file presence and validity

The result shows a clear verdict (ready / degraded / broken) with exact next steps for any issues.

Your First Test Case

Once /ado-check shows "ready", here's the workflow:

  1. Draft — Type /qa-draft and provide a User Story ID. The MCP reads the full story context (fields, linked pages, images) and generates a test case draft with solution summary and QA cheat sheet in your workspace.
  2. Review — Open the generated files in your editor. Edit test cases, adjust coverage, fix any assumptions. The draft is yours to shape.
  3. Publish — Type /qa-publish with the same User Story ID. The MCP checks for duplicates, asks for confirmation, then creates the test cases in ADO and auto-organizes them into the correct suite hierarchy.

Troubleshooting

MCP server not showing in Cursor

Restart Cursor after install. Check that ~/.cursor/mcp.json contains an vortex-ado entry.

ADO connection fails

Verify your PAT hasn't expired. Check that you have the correct org and project names. Run /ado-check for a detailed diagnostic.

Confluence pages not loading

Ensure your Confluence API token is valid and the email matches. The system tries the direct URL first, then falls back to the Atlassian Cloud API automatically.
