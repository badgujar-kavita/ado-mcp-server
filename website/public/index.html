Why Vortex ADO

Your QA team deserves better than copy-paste.

Traditional QA test design means reading user stories in one tab, Confluence specs in another, and manually writing test cases in a third. Vortex ADO gives your AI agent the complete picture — so it drafts test cases that actually reflect the requirements.

01

Full-context awareness

The agent sees everything relevant to a user story — every configured rich-text field, every populated ADO field (noise filtered), linked Confluence page content, and embedded images as pixel data. No more "can you also check the acceptance criteria?"

02

Draft-review-publish workflow

Test cases land as reviewable Markdown in your workspace before anything touches ADO. Your team reviews, edits, and approves locally — then publishes with a single command. No accidental test cases in production.

03

Duplicate prevention built in

Before publishing, Vortex ADO checks for existing linked test cases and shows you what it found. You choose whether to add, replace, or cancel — the system never creates duplicates silently.

04

Team conventions, zero code changes

Title format, persona labels, sprint prefixes, test-plan routing, image handling — all configurable per team. Your conventions propagate to every generated test case without touching source code.

How it works

Three steps per user story.

Once connected to your Azure DevOps organization, the per-story rhythm is simple.

Draft

Point the agent at a user story ID. It fetches the full context — requirements, acceptance criteria, solution design, linked documentation, and embedded visuals — then drafts comprehensive test cases into your local workspace.

Review

Open the generated Markdown in your editor. Test steps, prerequisites, and expected results are all human-readable. Your team reviews and refines before anything is committed to Azure DevOps.

Publish

One command pushes approved test cases to ADO. The system checks for duplicates, asks for explicit confirmation, creates the work items, links them to the user story, and writes the new IDs back into your draft for traceability.

Capabilities

What the agent has access to.

Vortex ADO is not a thin wrapper around a REST API. It provides structured, purpose-built context that lets the AI agent reason about your requirements instead of parsing raw JSON.

  • Full-context payload — returns every configured rich-text field, every populated ADO field (noise filtered), linked Confluence page content, and surfaces unfetched links (SharePoint, Figma) before drafting starts.
  • Vision into embedded images — the agent sees wireframes, flow diagrams, and UI mocks embedded in ADO fields and Confluence pages as actual pixel data, not just metadata links.
  • Test plan and suite management — navigate your project's test plan hierarchy, create sprint-based suite structures, and organize test cases where your team expects them.
  • Clone and adapt — copy test cases from one user story to another with intelligent classification. Each case is marked as clone-as-is, minor-update, or enhanced based on target context.
  • Structured output for MCP clients — every read command emits typed structured data alongside prose. Modern MCP clients render navigable trees; older clients see clean text.
  • Confluence integration — linked solution design pages are automatically fetched and included in the context. The agent reads your documentation so you do not have to re-explain it.

Trust and safety

Guardrails where they matter.

Vortex ADO runs entirely on your machine. Nothing phones home — not even version checks. Every request hits Azure DevOps or Confluence and nothing else.

Explicit consent required

Every action tool gates on an unambiguous "yes" before mutating. Frustration, sarcasm, and rhetorical questions are not treated as authorization — the agent re-asks with a clear yes/no prompt.

Credentials stay local

Your ADO Personal Access Token and optional Confluence API token are stored on your machine only. They are never committed, never synced, and never sent anywhere except the respective API endpoints.

No telemetry, no analytics

Zero tracking, zero phone-home, zero version pings. The server only communicates with Azure DevOps, Confluence, and your local filesystem.

Path-traversal protection

File writes use validated inputs with sanitized filenames. Every draft save and attachment download is verified against a known-safe workspace prefix.

Ship-dark by default

Advanced capabilities like image vision and local image saving are off by default. Teams opt in explicitly — no surprise behavior changes on install.

Schema-validated responses

Every ADO and Confluence API response passes through strict schema validation. Unexpected shapes surface as typed errors — never silent data loss in your test cases.
