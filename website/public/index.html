What makes ado-mcp different

Context the agent can actually use.

Most MCPs hand the agent a bare JSON blob and hope for the best. ado-mcp walks the user story, fetches linked Confluence pages, captures embedded images as pixel data, and returns a typed payload — every field the agent needs to draft a test case, in one structured response.

👁

Vision into rich-text

The agent actually sees images embedded in ADO work-item fields and Confluence attachment pages — wireframes, flow diagrams, Salesforce UI mocks. Not metadata, pixel data. Ship-dark behind images.returnMcpImageParts.

🧭

Full-context payload

get_user_story returns namedFields (every configured rich-text field), allFields (every populated ADO field with system-noise filter), fetchedConfluencePages[], and unfetchedLinks[] — SharePoint / Figma surfaced before drafting, not discovered halfway through.

🛡

Duplicate-TC preflight

When you /qa-publish to a US that already has linked test cases, the MCP shows a counts-based A/B/C prompt before doing anything. No more accidentally creating 8 duplicates.

🤝

Consent rule built in

Ambiguous replies ("are you dumb", "seriously?", emoji, silence) do not authorize tool calls. The agent re-asks with yes/no. Frustration isn't consent — the rule lives in AGENTS.md.

🎛

Per-team config

conventions.config.json customizes title format, persona labels, sprint prefix, test-plan mapping, image guardrails, and per-project custom fields. Your team's conventions, without code changes.

🧱

structuredContent on every read

All 12 read tools emit typed structured output alongside prose. MCP clients that support structuredContent render navigable trees; legacy clients see the exact same text as before. No regression, just upside.

🔐

PAT auth, credentials on-box

ADO Personal Access Token, Confluence API token — stored in ~/.ado-testforge-mcp/credentials.json on your machine. Nothing phones home.

🧩

21 slash commands, purpose-built

ado-* for ADO primitives, qa-* for the QA lifecycle, confluence-* for Confluence. Every command is a prompt with a contract; read tools never mutate, action tools gate on explicit yes.

All 21 commands

Everything this MCP does.

Twenty-one slash commands, grouped into setup, read, and action tools. Read tools are safe to run any time — they never modify ADO, Confluence, or your workspace. Action tools gate every mutation behind an explicit affirmative.

Setup

/ado-connect

Open the config UI. Enter ADO PAT, org, project, and optional Confluence token with real-time connection testing. Writes to ~/.ado-testforge-mcp/credentials.json.

Read

/ado-check

Deterministic setup-status check. Renders a Markdown table, overall verdict (ready / degraded / broken), and exact next actions. No agent hallucination.

Read

/ado-story

Fetch a User Story with full context: primary fields, allFields, fetchedConfluencePages, embeddedImages, unfetchedLinks. Emits structured output.

Read

/ado-plans

List every test plan in the project. ID, name, area path, state, root suite.

Read

/ado-plan

Get one test plan by ID. Area path, iteration, root suite — plus any attached query.

Read

/ado-suites

List suites under a plan. Root vs child relationships surfaced so the agent sees the real tree.

Read

/ado-suite

Get one suite by ID. The WIQL query string is surfaced as an artifact for query-based suites.

Read

/ado-suite-tests

List test cases inside a suite. IDs, titles, and links ready for the agent to reason over.

Read

/ado-fields

List every work-item field definition the project has configured. Capped at 50 in structured output; the full list is in prose.

Read

/qa-tests

List test cases linked to a User Story via the TestedBy relation. Numbered list with clickable web URLs.

Read

/qa-tc-read

Get one test case by ID. Relations are split into children (work-items) and artifacts (attachments) so the agent can pick the right bucket.

Action

/qa-draft

Generate US_<id>_test_cases.md, solution summary, and QA cheat sheet under tc-drafts/US_<id>/. Asks to confirm before writing; reads the full US context first.

Action

/qa-publish

Push an approved draft to ADO. Runs the duplicate-TC preflight, shows an A/B/C menu if linked test cases already exist, waits for explicit yes before mutating.

Action

/qa-tc-update

Update fields on an existing test case — title, description, prerequisites, steps, priority. Structured input preferred over raw HTML.

Action

/qa-tc-delete

Delete a single test case. Sends to Recycle Bin by default; pass destroy: true for permanent removal.

Action

/qa-tc-bulk-delete

Delete multiple test cases by ID. Confirmation-gated — the agent shows the full list and waits for yes.

Action

/qa-clone

Clone test cases from a source US to a target US. Analyzes target context, classifies each TC (clone-as-is / minor-update / enhanced), previews the plan before committing.

Action

/qa-suite-update

Update an existing test suite's name, parent, or query. Preview-then-confirm.

Action

/qa-suite-delete

Delete a test suite. The test cases themselves remain in ADO — only the suite association is removed.

Action

/qa-suite-setup-auto

Auto-derive plan + sprint from US AreaPath + Iteration and ensure the Sprint → Parent-US → US suite hierarchy. US ID only, nothing else.

Read

/confluence-read

Read a Confluence page by ID. Solution Design enrichment also auto-fetches when a US's Solution Notes field contains a Confluence URL.

Per-story flow

Three steps per story.

Once /ado-connect is done, your per-story rhythm looks like this.

Draft

Run /qa-draft <US-ID>. The MCP fetches the user story (all fields, linked Confluence pages, embedded images), then drafts test cases plus a solution summary and QA cheat sheet into tc-drafts/US_<id>/.

Review

Open the draft Markdown in your editor. Tables, prerequisites, and steps are all inline. Reviewers can click straight through to the original ADO work item and Confluence URLs without leaving the editor.

Publish

Run /qa-publish <US-ID>. The MCP checks for duplicates, shows a preflight if any are found, waits for explicit yes, calls the Azure DevOps REST API, and writes the new ADO IDs back into your draft.

Configure your team's conventions

Opinionated defaults, overridable per team.

Test-case title format, personas, sprint prefix, test-plan mapping, image guardrails — all configurable in one file. Drop conventions.config.json into your workspace and every command picks it up.

conventions.config.json — excerpt › 
{
  "testCaseTitle": {
    "prefix": "TC",
    "separator": " -> ",
    "template": "{prefix}_{usId}_{tcNumber}{separator}{featureTags}{separator}{summary}",
    "maxLength": 256
  },
  "prerequisiteDefaults": {
    "personas": {
      "KAM": {
        "label": "Key Account Manager (KAM) User",
        "roles": "KAM",
        "psg": "TPM Global KAM Users PSG"
      }
    },
    "personaRolesLabel": "TPM Roles",
    "personaPsgLabel": "PSG"
  },
  "suiteStructure": {
    "sprintPrefix": "SFTPM_",
    "testPlanMapping": [
      { "planId": 1066479, "areaPathContains": ["DHub", "D-HUB"] },
      { "planId": 1066480, "areaPathContains": ["EHub", "E-HUB"] }
    ]
  },
  "additionalContextFields": [
    { "adoFieldRef": "Custom.ImpactAssessment", "label": "Impact Assessment", "fetchLinks": true, "fetchImages": true }
  ],
  "images": {
    "enabled": true,
    "maxPerUserStory": 20,
    "returnMcpImageParts": false,
    "saveLocally": false
  }
}
  • testCaseTitle — format template plus length cap so every TC matches your naming convention.
  • prerequisiteDefaults.personas + personaRolesLabel / personaPsgLabel — your team's role terminology baked into every draft.
  • suiteStructure.sprintPrefix + testPlanMapping — your project's sprint pattern and the per-area-path plan routing.
  • additionalContextFields — extra ADO fields passed through to the agent alongside the primary allowlist.
  • images.returnMcpImageParts — the ship-dark vision flag. Off by default; flip it when you're ready.

Full schema documented in docs/setup-guide.md.

Built for trust

Guardrails where they matter.

Nothing about this MCP calls home, not even for version checks. Every request hits Azure DevOps or Confluence — nothing else — and every mutation waits for an affirmative token before it fires.

🤝

Consent rule

AGENTS.md § What counts as consent gates every tool call on an affirmative token. Frustration, sarcasm, rhetorical questions do not authorize action — the agent re-asks.

🔑

Local-only credentials

ADO PAT and optional Confluence token live in ~/.ado-testforge-mcp/credentials.json. Never committed, never synced, never sent anywhere except the respective Atlassian / Microsoft API endpoints.

🧱

Path-traversal safe

File writes use path.join() with Zod-validated inputs and a sanitizeFilename() guard. Every attachment download and draft save is verified against a known-safe workspace prefix.

🌙

Ship-dark defaults

Big-delta features — image vision, saving images to disk — default OFF. Teammates installing the same tarball see zero behavior change until they flip the flag themselves.

🚫

No telemetry

Zero analytics, zero phone-home, zero version pings. The MCP only talks to Azure DevOps, Confluence, and your local filesystem.

🧪

Zod-validated responses

Every ADO and Confluence response passes through a Zod schema. Unexpected shapes surface as typed errors — never silent data loss downstream.
